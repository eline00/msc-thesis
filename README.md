# ETC — Extract Test Commit

ETC is a CLI tool that proactively splits a tangled git commit into atomic, build-verified
commits.

It is one of the core contributions of an MSc thesis at DTU, _"Automatic Atomisation of Agentic Code
Changes"_, addressing:

## How it works

1. Snapshot the working-tree changes on a temporary `detangling` branch and record a `full.patch`
   (`git diff -U0`, zero context).
2. Split the patch into one file per hunk (`hunks/`).
3. Run **ddmin** ([`group.py`](autocommit/group.py)) over the remaining hunks to find the minimal
   subset that builds (`dotnet build --no-restore` is the oracle).
4. Commit that subset as one atomic commit (message generated by an LLM — see below).
5. Remove the committed hunks and repeat until none remain, or until an iteration makes no progress
   (the remaining hunks are unresolvable on their own).
6. Fast-forward the original branch over `detangling` and append run metrics.

Each `git apply --check` / `dotnet build` is one **invocation** — the primary cost metric.

### Post-hoc dependency analysis

Two read-only analyses run against a finished run directory and emit ordered patch bundles:

- `--analyze-build-deps` — probes **hard build dependencies** between atoms by combine-and-build
  ([`build_dep_analysis.py`](autocommit/build_dep_analysis.py)), iterating to a fixpoint, and writes
  merged components to `build_graph/`. Requires `--run-dir` and a clean tree checked out at the
  commit _before_ the first atom.

## Setup

Requires Python ≥ 3.11, [uv](https://docs.astral.sh/uv/), and the .NET SDK (`dotnet`) for the build
oracle.

```bash
uv sync                       # install into .venv, exposes the `etc` command
echo "GROQ_API_KEY=sk-..." > .env   # optional; see below
```

The `etc` console script is defined in [`pyproject.toml`](pyproject.toml). Run it with `uv run etc`
(or activate `.venv` and call `etc`).

**Commit messages** are generated by Groq (`llama-3.3-70b-versatile`) from the group's diff. If
`GROQ_API_KEY` is missing or the call fails, ETC falls back to a template message — the tool works
without a key.

## Usage

Run inside the target repository (the C# project you want to detangle), with the tangled changes in
the working tree. The default invocation runs the whole pipeline end-to-end:

```bash
uv run etc                            # detangle current working-tree changes
uv run etc dd25e301a-5hunks           # replay a benchmark case from autocommit/tests/
```

### Step-by-step

Each stage can be run individually — useful for inspecting a group before committing it. State is
persisted in `autocommit/run_state.json` between steps.

```bash
uv run etc --setup [test_name]    # create detangling branch, snapshot state, split hunks
uv run etc --next-iteration       # determine remaining hunks for this iteration
uv run etc --find-group           # run ddmin to find a minimal buildable group
uv run etc --commit-group         # commit the found group
uv run etc --one-iteration        # next-iteration + find-group + commit-group in one go
uv run etc --merge                # fast-forward original branch, write results
```

### Analysis

```bash
uv run etc --analyze-deps --run-dir <run-dir>        # def-use graph → graph/
uv run etc --analyze-build-deps --run-dir <run-dir>  # build deps → build_graph/
```

If interrupted (Ctrl+C), ETC resets to the tangled snapshot, restores the original branch, and
deletes `detangling`.

## Where the results live

### Per-run artifacts

A run of test case `<name>` writes to `autocommit/tests/<name>/runs/<run_id>/`, where `run_id` is a `YYYYMMDD_HHMMSS` timestamp:

```
runs/<run_id>/
├── hunks/          hunk_0001.patch, …   one file per hunk of the tangled diff
├── groups/         group_0001.patch, …  one file per committed atomic group
└── build_graph/    build_graph_NNNN.patch + build_dep_graph.log (--analyze-build-deps)
```

### Aggregated results ([`autocommit/results/`](autocommit/results/))

Appended once per completed run:

- [`runs.csv`](autocommit/results/runs.csv) — one row per run: `run_id`, `approach`, `repo`,
  `tangled_sha`, `total_hunks`, `groups_produced`, `total_invocations`, `total_duration_ms`,
  `build_cmd`, …
- [`groups.csv`](autocommit/results/groups.csv) — one row per produced group: `run_id`,
  `group_num`, `hunk_count`, `hunks`.

### Logs ([`autocommit/logging/`](autocommit/logging/))

- `run.log` — human-readable log of the most recent run.
- `metrics.log` — machine-readable `timestamp|event|data` events (`RUN_START`, `ITER_START`,
  `ITER_GROUP`, `RUN_END`, …)

[`autocommit/analysis.ipynb`](autocommit/analysis.ipynb) reads these to compare approaches across
runs.

## Benchmark & evaluation

Cases live under [`autocommit/tests/`](autocommit/tests/), named `<sha-prefix>-<N>hunks`. Each holds:

- `tangled.patch` — a synthetic tangled commit built from real atomic commits, and
- `gt-commitN.patch` — the ground-truth patch per concern, plus past `runs/`.

The synthetic tangled commits are constructed from real atomic commits in the
`commandlineparser/commandline` C# repository (the Flexeme benchmark). Grouping accuracy is measured
against ground truth.

---

## Scope

C# only (matching the Flexeme benchmark). `ddmin.py` is vendored directly from
`andrewchambers/ddmin-python`; the LLM is used only for commit-message text.
